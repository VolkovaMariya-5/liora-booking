Capstone-проект: Fullstack-приложение
Next.js (Frontend) + NestJS (Backend API) + PostgreSQL

1. Общее описание проекта
Capstone-проект — это финальная самостоятельная работа, в которой студент
разрабатывает полноценное fullstack-веб-приложение с нуля. Проект
демонстрирует умение проектировать архитектуру, писать frontend и backend-код,
работать с базой данных и деплоить готовый продукт.

2. Функциональные требования (MVP)
MVP (Minimum Viable Product) — минимально рабочая версия продукта, которую
студент обязан реализовать для получения зачёта. Все пункты ниже являются
обязательными.

2.1 Аутентификация и авторизация
• Регистрация нового пользователя (email + пароль)
• Вход в систему (email + пароль)
• Выход из системы
• Защита приватных маршрутов — неавторизованный пользователь
перенаправляется на страницу входа
• Роли пользователей: минимум 2 роли (например: USER и ADMIN)
• Хранение пароля в хешированном виде
  Аутентификация реализуется на стороне NestJS через Passport.js + JWT (пакеты
@nestjs/passport, passport-local, passport-jwt). OAuth (Google, GitHub) — по желанию, за
бонусные баллы.

2.2 CRUD основной сущности
Студент реализует полный CRUD (Create, Read, Update, Delete) для главной
сущности своего проекта:
• Создание записи через форму на frontend
• Отображение списка записей с пагинацией (минимум 10 записей на
страницу)
• Детальная страница одной записи
• Редактирование записи (только автор или ADMIN)
• Удаление записи с подтверждением (только автор или ADMIN)
• Поиск / фильтрация по полям (минимум 1 параметр)

2.3 NestJS Controllers (Backend API)

NestJS REST-контроллеры. Каждый контроллер соответствует одному модулю
(AuthModule, ProductsModule и т.д.). Минимально требуемые эндпоинты:

2.4 Swagger / OpenAPI — документация API
Каждый студент обязан задокументировать своё API с помощью Swagger (OpenAPI
3.0). Документация должна быть доступна по маршруту /api/docs в задеплоенном
приложении.
Требования к документации
• Задокументированы ВСЕ обязательные API-маршруты (минимум 7
эндпоинтов)
• Для каждого маршрута указаны: краткое описание, параметры (query / path /
body), коды ответа (200, 201, 400, 401, 403, 404)
• Для POST/PUT маршрутов описаны схемы requestBody с примерами
значений
• Для защищённых маршрутов указана схема безопасности (Bearer JWT или
Cookie-сессия)
  Swagger UI открывается по адресу /api/docs NestJS-сервера. Все декораторы
@ApiTags, @ApiOperation, @ApiResponse, @ApiBearerAuth должны быть
расставлены.

• JWT_EXPIRES_IN — время жизни токена (например: 7d)
• PORT — порт сервера (по умолчанию 3001)

4.3 Качество кода
• TypeScript: строгий режим (&quot;strict&quot;: true в tsconfig.json), минимум 80%
покрытия типами
• ESLint: конфигурация @nestjs/eslint-config для backend, next/core-web-vitals
для frontend — код обоих проектов должен проходить линтинг без ошибок
• Именование: компоненты в PascalCase, функции/переменные в camelCase,
файлы компонентов в kebab-case
• Комментарии: сложная бизнес-логика должна быть прокомментирована на
русском или английском
• Нет console.log в продакшн-коде (допустимо только для отладки, перед
сдачей убрать)

4.4 Безопасность (обязательно)
• Пароли хранятся только в виде хеша (bcryptjs, saltRounds &gt;= 10)
• Все приватные API-маршруты проверяют наличие и валидность сессии
• SQL-инъекции невозможны (Prisma использует параметризованные
запросы)
• Валидация входных данных на сервере перед записью в БД
• CORS настроен в NestJS через app.enableCors({ origin:
process.env.CORS_ORIGIN }) с явным указанием разрешённых источников
